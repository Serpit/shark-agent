# 理念库

跨方向通用的方法论与信念。区别于:

- `findings/`:单篇文章的提炼,**单点证据**。
- `themes.md`:具体的出海方向候选,**做什么**。
- `risks.md`:反模式,**不要做什么**。
- `principles.md`:**怎么思考、怎么判断**——为方向选择和动作决策提供准则。

每条理念由多篇 finding 的反复印证聚合而来,自下而上;反过来,选择 theme 或启动试错时,要回头对照 principles 是否一致。


## 抄出 MVP——别在验证期搞创新

- **陈述**:在赛道未跑通之前,直接抄头部对标,**不追求差异化**。MVP 阶段唯一目标是验证"这件事走得通",差异化是跑通之后的事。
- **信心等级**:supported(两条独立证据,内容生意和工具型产品都印证)

## 先发再迭代——做出 v0.1 优于完美计划

- **陈述**:哪怕第一版"糙得像垃圾"也要发出来,真实反馈是迭代的唯一燃料。"看攻略 1000 遍,不如做一次";同样适用于"创业 v0.1"——先做最小验证(例如一周内能不能拿到第一笔小钱)而非反复打磨方案。
- **信心等级**:strong(三条独立证据,内容/创业/工具型产品三类场景都印证;且与"用数量博概率"形成上下游互补——后者解决"做几个 v0.1")
- **形成依据**:[youtube-ai-faceless-channel-sop.md](findings/youtube-ai-faceless-channel-sop.md)(作者纯手工硬更 7 天才等到第一个万播,然后才动手做自动化);[2026-04-do-it-yourself-mindset.md](findings/2026-04-do-it-yourself-mindset.md)(作者建议朋友先 1 周内验证"出来干能不能拿到第一笔小钱");[2026-04-wechat-miniprogram-ai-matrix.md](findings/2026-04-wechat-miniprogram-ai-matrix.md)(作者明确"先上线再优化",所有成功案例第一版都不完美——大壮壮妈的博物馆讲解一开始只有几个馆,六六的「我在」内测时一堆 bug,但都是先发了真实用户才知道往哪改)
- **适用边界**:在风险可控、单次失败成本低的场景成立;在涉及合规/品牌/重资金投入的场景需先做风控,不能盲发。
- **首次记录**:2026-04-27

## 自动化前先核算时间成本

- **陈述**:做任何提效投入前,先把"每个工序当前消耗多少时间"量化,反推哪里值得自动化。时间是绝对稀缺,不算时间成本就分不清"看似在迭代实则在打杂"。

## 卖解决方案,而非卖产品

- **陈述**:在出海/B2B 等成熟市场,只「报价 + 发货」已经被卷成红海;真正的溢价来自把产品打包成「解决方案」——产品 + 认证 + 本地化适配 + 服务/培训/远程支持,谈判时把价格谈判引导为价值确认。
- **信心等级**:hypothesis(单证据,但跨多种场景出现:小语种本地化 / 制造业配套 / 非洲枢纽都依赖这一思路)
- **形成依据**:[2026-04-foreign-trade-b2b-newbie-sop.md](findings/2026-04-foreign-trade-b2b-newbie-sop.md)(作者把三个机会窗口的核心都归结为「从中间商升级为解决方案提供商」,启动案例也是「灯具 + GCC 认证 + 伊斯兰美学设计 + 1 年远程支持」打包)
- **适用边界**:适用于客单价较高、决策链复杂的 B2B 出海;不适用于纯 C 端 SKU 站、流量型电商等比拼极致价格/物流的场景。
- **首次记录**:2026-04-28
- **最近更新**:2026-04-28

## 供应链优先于产品(B2B 出海特化)

- **陈述**:做 B2B 贸易,选品的起点不是「我要卖什么」,而是「我能稳定调动什么样的供应链」。先花主要精力筛选并绑定 1-2 家柔性工厂(支持小单快反多批次),再从工厂能力反推可做的产品赛道,而不是相反。
- **信心等级**:hypothesis(单证据)
- **形成依据**:[2026-04-foreign-trade-b2b-newbie-sop.md](findings/2026-04-foreign-trade-b2b-newbie-sop.md)(作者明确提出「评估供应链能力 → 锁定适配品类 → 验证市场匹配度 → 深化合作」的闭环,把工厂视为盟友而非低价比价对象)
- **适用边界**:适用于贸易公司/SOHO/小工厂代销模式;不适用于自有工厂/品牌方/数字产品/纯内容生意。
- **首次记录**:2026-04-28

## 先验需求再投产能——BRD 优于 IDE

- **陈述**:打开 IDE 之前,先用 3 小时把"有没有人愿意为此付钱"调研清楚;独立开发者 90% 失败首因是没调研。AI 编程让 MVP 从 2 个月压到 2 周,但调研 AI 替不了你——**写代码是最贵的赌博,调研是最便宜的投资**。3 小时足以做 70% 调研,够做 GO/NO-GO 判断;NO-GO 不是失败,是省下 3 个月。
- **信心等级**:strong(三条独立证据,且与既有"先发再迭代"形成时间序补完——验证之后才是 v0.1)
- **形成依据**:[2026-04-indie-saas-3h-brd-validation.md](findings/2026-04-indie-saas-3h-brd-validation.md)(作者 pngtrid 先写 2 周代码上线才发现市场被占满,后改为 2 天调研重定位 1 个月变现 5K;给出 6 步 BRD SOP 与 GO/NO-GO 框架);[2026-04-foreign-trade-b2b-newbie-sop.md](findings/2026-04-foreign-trade-b2b-newbie-sop.md)(作者 19 年被女鞋营销话术裹挟未做调研,9 万启动金 3 个月烧光,触底负债 60w);[2026-04-wechat-miniprogram-ai-matrix.md](findings/2026-04-wechat-miniprogram-ai-matrix.md)(作者明确"选需求半天足够筛出一个好方向,选错了怎么努力都白搭",并给出可执行的三标准——功能单一 / 场景清晰 / 无头部产品;实战路径=打开微信搜"检测/换算/查询"看现有产品多烂)
- **适用边界**:适用于商业方向选择/MVP 立项;**与"先发再迭代"互补不冲突**——本原则解决"该不该做",先发再迭代解决"决定做之后如何启动"。3 小时是首次破冰的数量级,完全陌生赛道实际需 1-2 天熟练后才能压到 3 小时
- **首次记录**:2026-04-28
- **最近更新**:2026-04-29

## 差评是金矿,3-4 星是产品需求文档

- **陈述**:做需求挖掘时,**用户怒火比用户夸赞值钱 100 倍**。不要看竞品官网,要看用户在哪里骂竞品。1 星看核心痛点,3-4 星看"差那一点"——那就是切入点;真痛点判定 = 同一痛点 ≥3 次 + 来自不同平台。"竞品知道问题但一直没改"就是你的护城河。
- **信心等级**:hypothesis(单证据但方法论极具体可执行,与"需求挖掘"画像权重高度对齐)
- **形成依据**:[2026-04-indie-saas-3h-brd-validation.md](findings/2026-04-indie-saas-3h-brd-validation.md)(给出三层平台 + 四路搜索 + 五步看法 SOP,案例从 G2/Reddit/Trustpilot 翻出 remove.bg 真实定价/额度痛点,30 分钟产出痛点频次表)
- **适用边界**:适用于成熟赛道(竞品已经存在,有公开评价数据);**不适用于全新品类**(没有竞品就没有差评可挖,得换成用户访谈/痛点观察)。要警惕单平台回音壁(Reddit 天然反付费),必须多平台交叉
- **首次记录**:2026-04-28
- **最近更新**:2026-04-28

## 用数量博概率——做 N 个 v0.1,胜过赌 1 个完美

- **陈述**:在 AI 把单产品试错成本压到"几天 + 几十元"之后,**矩阵打法成为可行策略**——不再赌某一个产品能否爆,而是快速做 5-10 个分散组合,接受 30-50% 废掉,用整体收益覆盖单点失败。这是**组合层面的确定性**,与"先发再迭代"配合(后者解决"如何启动单产品",本原则解决"启动几个产品")。
- **信心等级**:hypothesis(单证据,但方法论极具体可执行,且解释了"AI 让独立开发者打法改变"的根本机制——以前一次失败伤筋动骨,现在是普通成本)
- **形成依据**:[2026-04-wechat-miniprogram-ai-matrix.md](findings/2026-04-wechat-miniprogram-ai-matrix.md)(作者 15 个小程序实测:7 稳定 + 5 一般 + 3 废,失败成本约 200 元 vs 成功收入 2w/月;明确指出"AI 出现之前个人开发者不可能跑矩阵,因为单次失败成本太高")
- **适用边界**:**只在单产品试错成本已被技术压低的赛道成立**——AI 编程能覆盖的工具型产品(小程序、Chrome Extension、CLI、轻 SaaS)、内容矩阵、模板/数字产品。**不适用**于:需要长期供应链投入的实物贸易、需要监管资质的赛道(金融/医疗)、需要长期用户教育的行业垂直产品——这些场景单产品成本无法被 AI 压低,矩阵思维会变成"哪个都做不深"的发散陷阱。
- **风险提示**:容易被误解为"什么都试试"——错。矩阵的前提是"每个 v0.1 都基于 BRD 选需求三标准,不是凭感觉发散"。**矩阵不是放弃判断,是在已经过滤的方向里做组合下注**。
- **首次记录**:2026-04-29
- **最近更新**:2026-04-29

## 流程数据 > 代码——工具产品的真正护城河

- **陈述**:做工具产品时,**真正值钱的不是脚本/界面,而是产品在跑过程中沉淀下来的"流程数据"**——平台规则、失败原因、成功率、踩坑记录、领域 know-how。代码可以重写、UI 可以复刻,但平台经验和失败记录很难凭空生成。这与工业软件值钱的逻辑同构(工业软件贵的不是代码,是行业 know-how 的数字化沉淀)。
- **信心等级**:hypothesis(单证据,但与"卖解决方案而非卖产品"形成方向呼应)
- **形成依据**:[2026-04-claude-code-sideline-100k.md](findings/2026-04-claude-code-sideline-100k.md)(作者半自动化外链系统的核心价值不是 Playwright 脚本,是 PostgreSQL 里几千个平台的流程库:平台类型/提交入口/审核规则/成功率/失败原因——把方法论 + Skill 包 + 流程库打包卖,就是"流程数据"在变现而非"脚本"在变现)
- **适用边界**:适用于**有重复流程 + 平台/规则差异 + 失败成本可被记录**的工具型产品(自动化/合规/审核/外链/选品/采购);**不适用**于纯算法产品、UI 创意产品、内容产品——它们的价值在算法/创意/内容本身而非流程沉淀。**实操含义**:做工具时要主动把"在跑过程中观察到的平台行为"以结构化字段沉淀(数据库表/CSV/Markdown 都行),而不是只把它们留在脚本注释里。
- **首次记录**:2026-04-29
- **最近更新**:2026-04-29

## 副业要做能离开人也跑的事——资产型 vs 卖时间

- **陈述**:副业的形态选择应优先**资产型**(SEO 站 / 内容矩阵 / 工具产品订阅 / 模板)而非**卖时间型**(辅导 / 定制项目 / 一次性咨询)。卖时间型即使单次客单高,**人不在收入就停**,且消耗时间挤压其他复利线;资产型即使单点收入小,具有"睡后收入"特征 + 不挤占其他线时间。前者天花板由"小时数 × 单价"决定,后者天花板由"系统跑通后的复制成本"决定。
- **信心等级**:supported(两条独立证据,且与既有"用数量博概率"形成形态层呼应——后者讲做几个,本条讲选什么形态)
- **形成依据**:[2026-04-claude-code-sideline-100k.md](findings/2026-04-claude-code-sideline-100k.md)(作者主动停掉抖音定制项目路径——抖音 7-8 个企业客资里只有 1 个真匹配,7w 项目按小时折算并不轻松,且持续 1-2 个月占用所有精力,期间无法推进 SEO/Skill 包/IP 三条复利线;原话"我更想做有复利的东西");[2026-04-wechat-miniprogram-ai-matrix.md](findings/2026-04-wechat-miniprogram-ai-matrix.md)(作者放弃公众号/抖音 DSO 等"持续发布持续推荐"路径,选择小程序矩阵——"做完不用管,半年还在赚")
- **适用边界**:适用于副业/兼职/独立开发者形态选择;**不适用**于早期需要靠服务现金流过渡的创业者(此时定制交付是必要的现金流来源,但要明确它是过渡性而非终态)。**操作含义**:面对一个变现机会要先问"这件事人不在还跑不跑";如果停了就停了,要警惕它对其他复利线的时间挤压。
- **首次记录**:2026-04-29
- **最近更新**:2026-04-29

## 需求发现路径要匹配产品分发生态

- **陈述**:**需求发现的"信号源"必须与产品的"分发生态"匹配,否则找到的痛点用户不会迁移到你的产品形态**。Web/SaaS 产品 ← Google 关键词搜索量 + SERP 缺口 + AnswerThePublic;App 产品 ← App Store 评论 + TestFlight + 应用内反馈;Chrome Extension ← Chrome Store 评论 + Reddit r/chrome;社群型产品 ← 平台内搜索 + 评论 + 收藏(对应 [community-demand-discovery.md](methods/community-demand-discovery.md))。**反例**:用 App 差评挖到的痛点拿去做 Web 工具——这些用户已经习惯在 App 生态里找解决方案,不会迁移到浏览器侧。
- **信心等级**:hypothesis(本项目用户教练对话中沉淀的判断,2026-04-29;尚未跑 MVP 验证,但与"差评是金矿"+"先验需求再投产能"形成正交补充——前两者讲"怎么挖痛点",本条讲"在哪挖才不浪费")
- **形成依据**:用户在选择 BRD 候选时主动指出:"从 app 里的差评找到的需求,基本还是在 app 生态里面,不会转移到网页侧"——这是 dogfood 判断,不来自 finding 但与 [methods/community-demand-discovery.md](methods/community-demand-discovery.md) 形成方法论对位
- **适用边界**:适用于工具型/SaaS/内容产品的**需求发现阶段**;赛道已验证、做迭代优化时可混用多源信号(差评 + 关键词 + 用户访谈),不必拘泥单一信号源。**实操含义**:启动 BRD 前先确定产品形态,再选对应的需求挖掘 SOP,不要倒过来"先有候选再选 SOP"
- **首次记录**:2026-04-29
- **最近更新**:2026-04-29

## 不要裸辞创业——心态急是反模式

- **陈述**:在没有验证项目方向的情况下裸辞,会陷入「越急 → 越乱选 → 越亏 → 更急」的负循环;每个项目都现学现卖、1-2 个月浅尝辄止,本质是心态失衡而非方向错误。试错应在保留收入或现金流缓冲的前提下进行。
- **信心等级**:supported(作者两次创业期亲历,且明确以反例告诫;也与「自动化前先核算时间成本」形成同源警示——稀缺感会扭曲决策)
- **形成依据**:[2026-04-foreign-trade-b2b-newbie-sop.md](findings/2026-04-foreign-trade-b2b-newbie-sop.md)(作者从工作辞职后 SOHO 2.0 阶段「一直尝试很多项目,一直都不赚钱,换得很勤快」,亲承裸辞心态会比较急)
- **适用边界**:适用于个人/小团队创业启动决策;若已有充足现金流(覆盖 12+ 个月生活成本)且方向已小范围验证,则不构成约束。
- **首次记录**:2026-04-28
- **最近更新**:2026-04-28
